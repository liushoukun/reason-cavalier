# 系统化调试

## 概述

随机打补丁浪费时间并制造新 bug。快速遮盖掩盖根因。

**核心原则：** **永远**先找根因再谈修复。对症修复是失败。

**违反本流程的字面要求，就是违反调试的精神。**

## 铁律

```
未完成第 1 阶段根因调查，不得提出任何修复
```

若你未完成第 1 阶段，就不能提议修复。

## 何时使用

适用于**任何**技术问题：
- 测试失败  
- 生产 bug  
- 意外行为  
- 性能问题  
- 构建失败  
- 集成问题  

**尤其**在以下情况使用：
- 时间压力（紧急时更容易猜）  
- 「明显一个小修」很诱人  
- 你已试过多种修复  
- 上次修复无效  
- 你并未完全理解问题  

**不要在以下情况跳过：**
- 问题看起来简单（简单 bug 也有根因）  
- 你很赶（慌乱保证返工）  
- 经理要「立刻修好」（系统化比乱试更快）  

## 四个阶段

**必须**完成每一阶段再进入下一阶段。

### 第 1 阶段：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**  
   - 不要跳过错误或警告  
   - 常含精确答案  
   - 读完整栈、行号、路径、错误码  

2. **稳定复现**  
   - 能否可靠触发？  
   - 精确步骤？  
   - 是否每次都发生？  
   - 若无法复现 → 收集更多数据，不要猜  

3. **检查近期变更**  
   - 什么变更可能导致？  
   - git diff、近期提交  
   - 新依赖、配置变更  
   - 环境差异  

4. **多组件系统中收集证据**  

   **当系统有多层（CI → 构建 → 签名，API → 服务 → 数据库）：**  

   **在提议修复前，先加诊断埋点：**  
   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传递
     - 检查每层状态

   运行一次收集证据，定位**在哪一层断裂**
   再分析证据锁定故障组件
   再深入该组件调查
   ```

5. **追踪数据流**  

   **当错误深埋在调用栈：**  

   见本目录 `root-cause-tracing.md` 了解完整逆向追踪技术。  

   **速览：**  
   - 坏值从哪来？  
   - 谁用坏值调用了这里？  
   - 一路向上直到源头  
   - **在源头修复**，而非在症状处  

### 第 2 阶段：模式分析

**修复前先找模式：**

1. **找工作正常的相似代码**  
2. **对照参考实现** — 若实现某模式，**完整**阅读参考，不要扫一眼  
3. **列出工作版与损坏版的所有差异** — 再小也别假设「不可能有关」  
4. **理解依赖** — 需要哪些组件、配置、环境、隐含假设？  

### 第 3 阶段：假设与验证

**科学方法：**

1. **单一假设** — 写清：「我认为根因是 X，因为 Y」  
2. **最小验证** — 用**最小**变更验证假设，一次只动一个变量  
3. **继续前核实** — 有效 → 第 4 阶段；无效 → 新假设；**不要**叠罗汉式加修复  
4. **不懂就说不懂** — 「我不懂 X」、求助、再研究  

### 第 4 阶段：实现

**修根因，不是症状：**

1. **先建失败用例** — 最简单复现；自动化测试优先；必要时一次性脚本；**修复前必须有**；编写规范失败测试见 `superpowers:test-driven-development`  
2. **单一修复** — 针对已识别根因；**一次**一个变更；无「顺手」重构、无捆绑重构  
3. **验证修复** — 测试绿？其他测试仍绿？问题真解决？  
4. **若修复无效** — **停下**；数你已试过几次；**<3 次：** 回第 1 阶段用新信息重分析；**≥3 次：** **停下**质疑架构（见下）  
5. **≥3 次失败：质疑架构**  

   **可能表明架构问题的模式：**  
   - 每次修复在不同位置暴露新的共享状态/耦合  
   - 修复需要「大规模重构」  
   - 每次修复在别处制造新症状  

   **停下并与人类搭档讨论根本设计** — 这不是失败假设，而可能是错误架构。  

## 危险信号 — 停下并遵循流程

若你发现自己想：
- 「先 quick fix，以后再查」  
- 「先改 X 试试」  
- 「一次改多处再跑测试」  
- 「跳过测试，我手动验」  
- 「大概是 X，我修一下」  
- 「不完全懂但这可能有用」  
- 「模式说 X 但我换个做法」  
- 「主要问题：[列修复]」却**未**追踪数据流  
- **「再试一次修复」（已试过 2+）**  
- **每次修复在不同地方暴露新问题**  

**以上任一条 = 停下。回第 1 阶段。**  
**若 ≥3 次修复失败：** 质疑架构（见第 4 阶段第 5 点）

## 人类搭档提示「你走偏了」

- 「是不是没发生？」— 你在未核实前就假设  
- 「能不能看到…？」— 你本该加证据采集  
- 「别猜」— 你在未理解前提修复  
- 「Ultrathink」— 质疑根本，而非仅症状  
- 「我们是不是卡住了？」（沮丧）— 你的方法行不通  

**看到这些：** 停下。回第 1 阶段。

## 常见自我合理化

| 借口 | 事实 |
|------|------|
| 「问题简单不需要流程」 | 简单问题也有根因。流程对简单 bug 也很快。 |
| 「紧急没时间走流程」 | 系统化比猜-试错更快。 |
| 「先试这个再调查」 | 第一次修复定调。一开始就做对。 |
| 「确认修复后再写测试」 | 未测修复站不住。先有失败测试证明。 |
| 「一次改多处省时间」 | 无法隔离何者生效。会引入新 bug。 |
| 「参考太长我按理解改」 | 一知半解必出 bug。要读全。 |
| 「我看到问题了让我修」 | 看到症状 ≠ 理解根因。 |
| 「再试一次」（已失败 2+） | 3+ 失败常是架构问题。质疑模式，别再盲修。 |

## 速查

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| **1. 根因** | 读错误、复现、查变更、收集证据 | 理解 **是什么** 与 **为什么** |
| **2. 模式** | 找正常示例、对比 | 找出差异 |
| **3. 假设** | 立论、最小验证 | 证实或新假设 |
| **4. 实现** | 建测试、修复、验证 | Bug 解决、测试通过 |

## 当流程显示「无根因」

若系统调查后问题确属环境、时序或外部：

1. 你已走完流程  
2. 记录你调查过什么  
3. 实现合适处理（重试、超时、错误提示）  
4. 增加监控/日志供日后查  

**但：** 声称「无根因」的案例里约 95% 是调查不完整。

## 支撑技术

本目录内属于系统化调试的技术：

- **`root-cause-tracing.md`** — 沿调用栈逆向找原始触发  
- **`defense-in-depth.md`** — 找到根因后在多层加校验  
- **`condition-based-waiting.md`** — 用条件轮询替代任意超时  

**相关技能：**  
- **superpowers:test-driven-development** — 第 4 阶段第 1 步写规范失败测试  
- **superpowers:verification-before-completion** — 声称修复成功前先核实  

## 实际影响

来自调试会话：
- 系统化：约 15–30 分钟修复  
- 乱试：2–3 小时空转  
- 首次修复成功率：约 95% vs 40%  
- 引入新 bug：几乎为零 vs 常见
