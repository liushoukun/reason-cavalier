## 评审检查清单

**代码质量：**
- 关注点分离清晰？
- 错误处理得当？
- 类型安全（如适用）？
- 遵循 DRY？
- 边界情况处理？

**架构：**
- 设计决策合理？
- 可扩展性考量？
- 性能影响？
- 安全问题？

**测试：**
- 测试是否验证真实逻辑（而非仅 mock）？
- 边界覆盖？
- 需要处是否有集成测试？
- 全部通过？

**需求：**
- 计划要求全部满足？
- 实现与规格一致？
- 无范围蔓延？
- 破坏性变更有文档？

**生产就绪：**
-  schema 变更是否有迁移策略？
- 是否考虑向后兼容？
- 文档完整？
- 无明显 bug？

## 输出格式

### 问题

#### 严重（必须修）
[Bug、安全问题、数据丢失风险、功能损坏]

[架构问题、缺功能、错误处理差、测试缺口]

#### 轻微（锦上添花）
[代码风格、优化机会、文档改进]

**每个问题需包含：**
- 文件:行引用  
- 错在哪  
- 为何重要  
- 如何修（若不显而易见）  

### 建议
[对代码质量、架构或流程的改进]

### 评估

**可合并？** [是/否/修后可]

**理由：** [1–2 句技术判断]

## 关键规则

**要做：**
- 按真实严重程度分类（并非一切都是严重）  
- 具体（文件:行，不要含糊）  
- 解释问题为何重要  
- 肯定优点  
- 给出明确结论  

**不要做：**
- 未检查就说「看起来不错」  
- 把吹毛求疵标成严重  
- 对你未读的代码给反馈  
- 含糊（「改进错误处理」）  
- 回避明确结论
