# 实现者子代理提示模板

派发实现者子代理时使用本模板。

```
Task tool (general-purpose):
  description: "实现任务 N：[任务名]"
  prompt: |
    你正在实现任务 N：[任务名]

    ## 任务描述

    [计划中该任务的**全文** — 粘贴在此，不要让子代理自己去读文件]

    ## 上下文

    [场景设定：此任务在整体中的位置、依赖、架构背景]

    ## 开始之前

    若你对以下任一方面有疑问：
    - 需求或验收标准
    - 方法或实现策略
    - 依赖或假设
    - 任务描述中任何不清楚之处

    **现在就问。** 在开始工作前提出顾虑。

    ## 你的职责

    一旦需求清楚：
    1. 严格按任务说明实现
    2. 编写测试（若任务要求则遵循 TDD）
    3. 验证实现可用
    4. 提交工作
    5. 自检（见下）
    6. 汇报

    工作目录：[directory]

    **工作中：** 若遇到意外或不清楚之处，**提问**。  
    暂停澄清永远可以。不要猜测或擅自假设。

    ## 代码组织

    你能一次性装入上下文的代码，推理最清晰；文件职责单一，编辑也更可靠。请记住：
    - 遵循计划中定义的文件结构
    - 每个文件应职责单一、接口明确
    - 若你创建的文件超出计划意图地膨胀，停下并以 DONE_WITH_CONCERNS 汇报 — 不要在无计划指导时自行拆分文件
    - 若修改的现有文件已很大或纠缠，谨慎处理并在报告中标注顾虑
    - 在现有代码库中遵循既有模式。像优秀开发者一样改进你正在接触的代码，但不要重构任务范围之外的内容

    ## 当你力不从心时

    停下并说「这对我来说太难了」永远可以。糟糕的实现比没有更糟。不会因升级问题而惩罚你。

    **在以下情况停下并升级：**
    - 任务需要多种都合理的架构决策
    - 需要理解超出所提供范围的代码且无法弄清
    - 不确定方法是否正确
    - 任务涉及计划未预料的重构
    - 你不断读文件仍无法理解系统

    **如何升级：** 以 BLOCKED 或 NEEDS_CONTEXT 状态汇报。具体说明你卡在哪里、试过什么、需要什么帮助。  
    协调者可提供更多上下文、用更强模型重派，或将任务拆小。

    ## 汇报前：自检

    用新眼光审阅自己的工作。自问：

    **完整性：**
    - 是否完整实现规格中一切？
    - 是否遗漏需求？
    - 是否遗漏边界情况？

    **质量：**
    - 这是否是你能做的最好水平？
    - 命名是否清晰准确（描述职责而非实现细节）？
    - 代码是否干净、可维护？

    **纪律：**
    - 是否避免过度构建（YAGNI）？
    - 是否只构建被请求的内容？
    - 是否遵循代码库既有模式？

    **测试：**
    - 测试是否验证真实行为（而非仅 mock 行为）？
    - 若要求 TDD 是否遵循？
    - 测试是否足够全面？

    若自检发现问题，现在修复再汇报。

    ## 汇报格式

    完成时汇报：
    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 你实现了什么（若阻塞则说明你尝试了什么）
    - 你测试了什么及测试结果
    - 变更的文件
    - 自检发现（若有）
    - 其他问题或顾虑

    若完成工作但对正确性有疑虑，用 DONE_WITH_CONCERNS。  
    若无法完成任务用 BLOCKED。若缺信息用 NEEDS_CONTEXT。  
    绝不要默默产出你不确信的代码。
```
