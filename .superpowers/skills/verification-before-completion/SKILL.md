# 完成前验证

未经验证就声称完成，不是高效，而是不诚实。

**核心原则：** 先有证据，再下结论。

**违反本规则的字面要求，就是违反其精神。**

## 铁律

```
没有本轮消息中的全新验证输出，就不得声称通过/完成
```

若你未在本轮消息中运行验证命令，就不能声称测试通过。

## 闸门函数

```
在声称任何状态或表达满意之前：

1. 识别：什么命令能证明该声称？
2. 运行：执行**完整**命令（全新、完整）
3. 阅读：完整输出、退出码、失败计数
4. 核实：输出是否支持该声称？
   - 若否：用事实陈述真实状态
   - 若是：连同**证据**一起陈述声称
5. 仅在此之后：作出声称

跳过任一步 = 撒谎，而非验证
```

## 常见失败

| 声称 | 需要 | 不足够 |
|------|------|--------|
| 测试通过 | 测试命令输出：0 失败 | 上次运行、「应该过」 |
| Lint 干净 | Lint 输出：0 错误 | 部分检查、外推 |
| 构建成功 | 构建命令：exit 0 | Lint 过、日志「看起来好」 |
| Bug 已修 | 复现原症状的测试通过 | 代码改了、假设已修 |
| 回归测试有效 | 已验证红-绿循环 | 测试只跑过一次通过 |
| 代理已完成 | VCS diff 显示变更 | 代理报告「成功」 |
| 需求已满足 | 逐条核对清单 | 仅测试通过 |

## 危险信号 — 停下

- 使用「应该」「大概」「似乎」  
- 验证前表达满意（「太好了！」「完美！」「搞定！」等）  
- 未验证就要提交/推送/PR  
- 轻信代理成功报告  
- 依赖部分验证  
- 想「就这一次」  
- 疲惫想赶快结束  
- **任何暗示成功却未运行验证的措辞**  

## 合理化预防

| 借口 | 事实 |
|------|------|
| 「现在应该好了」 | **去跑**验证 |
| 「我有信心」 | 信心 ≠ 证据 |
| 「就这一次」 | 没有例外 |
| 「Lint 过了」 | Lint ≠ 编译器 |
| 「代理说成功」 | 独立核实 |
| 「我累了」 | 疲惫不是借口 |
| 「部分检查够了」 | 部分证明不了什么 |
| 「换说法就不算」 | 精神重于字面 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] 「全部测试通过」
❌ 「现在应该过了」/「看起来对」
```

**回归测试（TDD 红-绿）：**
```
✅ 编写 → 运行（通过）→ 撤销修复 → 运行（**必须失败**）→ 恢复 → 运行（通过）
❌ 「我已写回归测试」（未做红-绿核实）
```

**构建：**
```
✅ [运行构建] [看到：exit 0] 「构建通过」
❌ 「Lint 过了」（Lint 不检查编译）
```

**需求：**
```
✅ 重读计划 → 建清单 → 逐项核实 → 报告缺口或完成
❌ 「测试过了，阶段完成」
```

**代理委派：**
```
✅ 代理报告成功 → 查 VCS diff → 核实变更 → 报告真实状态
❌ 轻信代理报告
```

## 为何重要

来自 24 条失败记忆：
- 人类搭档说「我不信你」— 信任破裂  
- 未定义函数上线 — 会崩溃  
- 未满足需求上线 — 功能不完整  
- 虚假完成浪费时间 → 返工  
- 违反：「诚实是核心价值。若你撒谎，会被替换。」  

## 何时适用

**在以下任何情况之前始终适用：**
- 任何成功/完成声称的变体  
- 任何满意表达  
- 任何对工作状态的正面陈述  
- 提交、创建 PR、任务完成  
- 进入下一任务  
- 委派给代理  

**规则适用于：**
- 原话、同义改写、成功暗示  
- 任何暗示完成/正确的沟通  

## 底线

**验证没有捷径。**

运行命令。读输出。**然后**再声称结果。

没有商量余地。
