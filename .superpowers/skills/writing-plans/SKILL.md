# 撰写实现计划

## 概述

撰写**全面**的实现计划，假设工程师对你们代码库**零上下文**且品味可疑。写清他们需要知道的一切：每个任务要动哪些文件、代码、测试、可能要查的文档、如何验证。把整个计划拆成可一口吃掉的小任务。DRY。YAGNI。TDD。频繁提交。

假设他们是熟练开发者，但几乎不了解你们的工具链或问题域，也不太懂良好测试设计。

**开始时宣告：**「我正在使用 writing-plans 技能来创建实现计划。」

**上下文：** 应在专用 worktree 中运行（由 brainstorming 技能创建）。

**保存计划到：** `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`  
- （用户对计划位置的偏好覆盖此默认）

## 范围检查

若规格覆盖多个独立子系统，应在头脑风暴阶段拆成子项目规格。若未拆，建议拆成**多个独立计划** —— 每个子系统一个计划。每个计划应能单独产出可运行、可测试的软件。

## 文件结构

在定义任务前，先列出将创建或修改的文件及各自职责。此处锁定分解决策。

- 设计边界清晰、接口明确的单元。每个文件应有一个明确职责。  
- 你能更好推理可一次性装入上下文的代码；文件聚焦时编辑更可靠。优先小而专的文件，而非大包大揽。  
- **一起变的文件应放在一起。** 按职责拆分，而非按技术分层。  
- 在现有代码库中遵循既有模式。若代码库惯用大文件，不要单方面大重构 —— 但若你正在修改的文件已难以维护，在计划中包含拆分是合理的。  

此结构指导任务分解。每个任务应产生相对独立、单独也说得通的变更。

## 任务粒度：一口大小

**每一步是一个动作（约 2–5 分钟）：**
- 「编写失败测试」— 一步  
- 「运行确认失败」— 一步  
- 「编写使测试通过的最小实现」— 一步  
- 「运行测试确认全绿」— 一步  
- 「提交」— 一步  

## 计划文档页眉

**每个计划必须以该页眉开头：**

```markdown
# [功能名] 实现计划

> **对代理工作者：** 必需子技能：使用 superpowers:subagent-driven-development（推荐）或 superpowers:executing-plans 按任务逐步实现。步骤使用复选框（`- [ ]`）语法跟踪。

**目标：** [一句话说明构建什么]

**架构：** [2–3 句说明思路]

**技术栈：** [关键技术/库]

---
```

## 任务结构

````markdown
### 任务 N：[组件名]

**文件：**
- 新建：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 测试：`tests/exact/path/to/test.py`

- [ ] **步骤 1：编写失败测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2：运行测试确认失败**

运行：`pytest tests/path/test.py::test_name -v`  
期望：FAIL，提示「function not defined」

- [ ] **步骤 3：编写最小实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4：运行测试确认通过**

运行：`pytest tests/path/test.py::test_name -v`  
期望：PASS

- [ ] **步骤 5：提交**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

## 禁止占位

每一步都必须包含工程师实际需要的具体内容。下列属于**计划失败** —— 永远不要写：

- 「TBD」「TODO」「稍后实现」「补细节」  
- 「添加适当错误处理」/「加校验」/「处理边界」  
- 「为上述编写测试」（却无具体测试代码）  
- 「与任务 N 类似」（请重复代码 —— 工程师可能乱序阅读任务）  
- 只描述做什么却不展示如何做（涉及代码的步骤必须有代码块）  
- 引用未在任何任务中定义的类型、函数或方法  

## 记住

- 始终给出**精确**文件路径  
- 每个变更步骤给出**完整**代码 —— 若步骤改代码，就要有代码块  
- 给出**精确**命令与期望输出  
- DRY、YAGNI、TDD、频繁提交  

## 自检

写完完整计划后，用新眼光对照规格自检 —— 这是你自己跑的清单，不是子代理。

**1. 规格覆盖：** 浏览规格每节/每条需求。能否指出实现它的任务？列出缺口。

**2. 占位扫描：** 在计划中搜索上文「禁止占位」中的红线模式。修复。

**3. 类型一致：** 后续任务中的类型、方法签名、属性名是否与前面任务一致？任务 3 叫 `clearLayers()` 任务 7 却叫 `clearFullLayers()` 就是 bug。

发现问题当场内联修复。无需整套重审 —— 修完继续。若某规格需求无对应任务，补任务。

## 执行交接

保存计划后，提供执行方式选择：

**「计划已完成并保存到 `docs/superpowers/plans/<filename>.md`。两种执行方式：**

**1. 子代理驱动（推荐）** — 每任务派发全新子代理、任务间评审、快速迭代  

**2. 本会话内联执行** — 在本会话用 executing-plans 执行，分批带检查点  

**选哪一种？」**

**若选子代理驱动：**
- **必需子技能：** superpowers:subagent-driven-development  
- 每任务新子代理 + 两阶段评审  

**若选本会话内联：**
- **必需子技能：** superpowers:executing-plans  
- 分批执行与审阅检查点
