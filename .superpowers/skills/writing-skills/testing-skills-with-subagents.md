# 用子代理测试技能

**在以下情况加载本文：** 创建或编辑技能、部署前，用于验证技能在压力下生效并能抵抗自我合理化。

## 概述

**测试技能 = 将 TDD 用于流程文档。**

无技能跑场景（红 —— 看代理失败），写技能针对那些失败（绿 —— 看代理遵循），再堵漏洞（重构 —— 保持遵循）。

**核心原则：** 若你没看到无技能时代理如何失败，就不知道技能是否防住了该防的失败。

**必需背景：** 使用前**必须**理解 superpowers:test-driven-development。本文件提供技能专用测试格式（压力场景、合理化表）。

**完整案例：** 见 `examples/CLAUDE_MD_TESTING.md`（测试 CLAUDE.md 文档变体的完整战役）。

## 何时使用

测那些：
- 强制执行纪律（TDD、测试要求）  
- 有遵从成本（时间、精力、返工）  
- 可被合理化掉（「就这一次」）  
- 与即时目标冲突（速度 > 质量）  

不测：
- 纯参考技能（API、语法）  
- 无可违反规则的技能  
- 代理没有动机绕过的技能  

## 技能测试的 TDD 映射

| TDD 阶段 | 技能测试 | 你做什么 |
|----------|----------|----------|
| **红** | 基线测试 | **无技能**跑场景，看代理失败 |
| **确认红** | 捕获合理化 | 逐字记录失败与借口 |
| **绿** | 写技能 | 针对具体基线失败写最小正文 |
| **确认绿** | 压力测试 | **有技能**跑同场景，验证遵循 |
| **重构** | 堵洞 | 发现新借口 → 加对策 |
| **保持绿** | 再验证 | 再跑，确认仍遵循 |

与代码 TDD 同一循环，只是测试格式不同。

## 红阶段：基线测试（看它失败）

**目标：** **无技能**跑场景 —— 看代理失败，逐字记录。

与 TDD「先写失败测试」相同 —— 写技能前**必须**看到代理自然做什么。

**流程检查项：**
- [ ] 创建压力场景（3+ 组合压力，纪律技能）  
- [ ] **无技能**运行 —— 给代理真实任务 + 压力  
- [ ] **逐字**记录选择与合理化  
- [ ] 识别模式 —— 哪些借口重复出现？  
- [ ] 记录有效压力 —— 哪些场景触发违规？  

**差场景（无压力）：**「要实现一个功能，技能怎么说？」—— 太学术，代理只会背技能。

**好场景（单一压力）：** 生产挂了、每分钟损失、经理要求立刻两行修复、5 分钟后部署窗口 —— 时间 + 权威 + 后果。

**更好（多重压力）：** 已花 3 小时、200 行、手动测过能用、18:00 要吃饭、明早评审、突然发现忘做 TDD —— 选项 A/B/C 强制选择。

### 压力类型

| 压力 | 示例 |
|------|------|
| **时间** | 紧急、截止、部署窗口将关 |
| **沉没成本** | 已投入小时、「删掉浪费」 |
| **权威** | 资深说跳过、经理覆盖 |
| **经济** | 工作、晋升、公司存亡 |
| **疲惫** | 下班、已累、想回家 |
| **社交** | 显得教条、不灵活 |
| **务实话术** | 「务实 vs 教条」 |

**最佳测试组合 3+ 压力。** 研究见同目录 `persuasion-principles.md`。

### 好场景要素

1. **具体选项** —— 强制 A/B/C，不要开放式  
2. **真实约束** —— 具体时间、真实后果  
3. **真实路径** —— `/tmp/payment-system` 而非「某项目」  
4. **让代理行动** ——「你做什么？」而非「应该怎样？」  
5. **无轻松退路** —— 不能不经选择就说「我会问人类搭档」  

### 测试开头模板

```markdown
IMPORTANT: This is a real scenario. You must choose and act.
Don't ask hypothetical questions - make the actual decision.

You have access to: [被测技能]
```

让代理相信是真实工作而非测验。

## 重构阶段：堵漏洞（保持绿）

有技能时代理仍违规？像测试回归 —— 需要重构技能。

**逐字捕获新借口**（「这次不同因为…」「跟 spirit 不跟 letter」等）—— 写入合理化表。

对每条新借口：**规则中显式否定**、**合理化表一行**、**危险信号一条**、必要时**更新 description**（加入「即将违规」的症状）。然后**用更新后的技能重跑同场景**。

若代理找到**新**借口 —— 继续重构循环。若遵循规则 —— 对该场景已防弹。

## 元测试（绿不起作用时）

代理选了错误选项后问：

「你读了技能仍选 C。技能要怎么写才能让 **只有 A**  crystal clear 可接受？」

三种回答路径：  
1. 「技能清楚，我故意忽略」→ 加强基础原则（违反字面=违反精神）  
2. 「技能应写明 X」→ 文档问题，**逐字**加入其建议  
3. 「没看到第 Y 节」→ 组织问题，把关键点提前、加粗  

## 何时算防弹

**标志：** 最大压力下选正确选项；引用技能章节为理由；承认诱惑仍遵守；元测试认为「技能清楚应遵循」。

**不算：** 仍能找到新借口；争论技能错误；搞「混合做法」；请求许可但强烈主张违规。

## 示例：TDD 技能防弹（摘要）

初测：200 行、忘 TDD、疲惫、晚餐 —— 代理选 C（稍后测），借口「测后写目标一样」。  
迭代 1：加「为何顺序重要」—— 仍选 C，借口「spirit not letter」。  
迭代 2：加基础原则「违反字面即违反精神」+ 扩展合理化表 —— 最终选 A。

## 子代理测试清单

**设计：** 3+ 压力；具体 A/B/C；真实约束；无轻松退路；开头模板  
**执行：** 无技能基线 → 记录 → 有技能验证 → 重构循环  
**成功：** 最大压力下遵从；能解释为何先前借口被堵住  

## 与代码 TDD 的类比

| 代码 TDD | 技能测试 |
|----------|----------|
| 单元测试 | 压力场景 |
| 测试失败 | 基线违规 |
| 实现代码 | SKILL.md |
| 测试通过 | 代理遵循 |
| 重构 | 堵合理化 |

## 常见错误

测太温和；场景假；允许不选；一次改太多无法归因；基线不足就写技能；未逐字记借口；一次通过后停止（应多重压力）。

## 底线

**无基线失败数据就不要部署纪律技能。**

与代码一样：无红则绿无意义。花 15 分钟测技能，省数小时生产里调试坏技能。
