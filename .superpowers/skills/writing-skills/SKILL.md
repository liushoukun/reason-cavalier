# 撰写技能

## 概述

**撰写技能 = 将 TDD 应用于流程文档。**

个人技能位于代理特定目录（Claude Code：`~/.claude/skills`；Codex：`~/.agents/skills/`）。

你编写测试用例（带子代理的压力场景），观察失败（基线行为），编写技能（文档），观察通过（代理遵循），再重构（堵漏洞）。

**核心原则：** 若你没看到**没有**技能时代理如何失败，就不知道技能是否教对了东西。

**必需背景：** 使用本技能前**必须**理解 superpowers:test-driven-development。该技能定义红-绿-重构循环；本技能把 TDD 适配到文档。

**官方参考：** Anthropic 官方技能撰写最佳实践见 `anthropic-best-practices.md`（本目录；与英文原文对应，可为摘要或全译）。本文档在 TDD 取向之外补充模式与准则。

## 什么是技能？

**技能**是关于成熟技术、模式或工具的参考指南，帮助未来的 Claude 找到并应用有效方法。

**技能是：** 可复用技术、模式、工具、参考手册  
**技能不是：** 讲你某一次如何解决问题的叙事  

## 技能的 TDD 映射

| TDD 概念 | 技能创建 |
|----------|----------|
| **测试用例** | 带子代理的压力场景 |
| **生产代码** | 技能文档（SKILL.md） |
| **测试失败（红）** | 无技能时代理违反规则（基线） |
| **测试通过（绿）** | 有技能时代理遵循 |
| **重构** | 堵漏洞同时保持遵循 |
| **先写测试** | **在写技能之前**跑基线场景 |
| **看它失败** | 逐字记录代理的合理化借口 |
| **最少文档** | 只针对那些具体违规写技能 |
| **看它通过** | 验证代理现已遵循 |
| **重构循环** | 发现新借口 → 堵上 → 再验证 |

整个技能创建过程遵循红-绿-重构。

## 何时创建技能

**创建当：**
- 技巧对你曾不直观  
- 你会在多个项目中再次参考  
- 模式适用范围广（非项目专属）  
- 他人也会受益  

**不要为以下创建：**
- 一次性解法  
- 别处已有良好文档的常规实践  
- 项目专属约定（放 CLAUDE.md）  
- 可用正则/校验强制执行的机械约束（应自动化；文档留给判断型决策）  

## 技能类型

- **技术（Technique）：** 有步骤的具体方法（如基于条件的等待、根因追踪）  
- **模式（Pattern）：** 思考问题的方式  
- **参考（Reference）：** API、语法、工具文档  

## 目录结构

```
skills/
  skill-name/
    SKILL.md              # 主参考（必需）
    supporting-file.*     # 仅在需要时
```

**扁平命名空间** — 所有技能处于同一可搜索空间。

**拆文件当：**
1. **重参考**（100+ 行）— API、全面语法  
2. **可复用工具** — 脚本、工具、模板  

**保留在内联：** 原则与概念、短代码模式（<50 行）、其余默认可内联。

## SKILL.md 结构

**YAML 头：**
- 必需字段：`name`、`description`（完整字段见 [agentskills.io/specification](https://agentskills.io/specification)）  
- 合计最多 1024 字符  
- `name`：仅字母、数字、连字符  
- `description`：**第三人称**，只写**何时使用**（不要写流程摘要）  
  - 以「在……时使用」式触发条件开头（与英文技能常用的 “Use when…” 对应）  
  - 包含具体症状、情境、上下文  
  - **绝不**总结技能流程（见下文 CSO）  
  - 可能时控制在 500 字符内  

正文建议含：概述、何时使用（可含小流程图）、核心模式、速查、常见错误、可选「实际影响」。

## Claude 搜索优化（CSO）

**发现至关重要：** 未来的 Claude 需要**找到**你的技能。

### 1. 丰富的 description

**用途：** Claude 用 description 决定此刻是否读该技能。要回答：「我现在该不该读这个？」

**关键：description = 何时用，不是做什么**

若 description 总结工作流，Claude 可能只读 description 而跳过正文。测试表明：写「任务间代码评审」会导致只做**一次**评审，尽管正文流程图显示**两阶段**（规格 + 质量）。改为仅触发条件后，Claude 会读流程图并遵循两阶段。

**陷阱：** 总结工作流的 description 会成为捷径；正文变成「可跳过文档」。

**内容：** 具体触发、症状、情境；描述**问题**（竞态、不稳定）而非仅语言症状（setTimeout）；技术无关除非技能本身技术相关；第三人称；**绝不**总结流程。

### 2. 关键词覆盖

使用 Claude 会搜索的词：错误信息、症状词、同义词、工具名、文件类型等。

### 3. 命名

主动、动词优先：`creating-skills` 而非 `skill-creation`；`condition-based-waiting` 优于泛名。

### 4. Token 效率

getting-started / 高频加载技能会进入许多会话 —— 每个 token 都重要。目标字数：getting-started 每工作流 <150 词；高频技能总计 <200 词；其他 <500 词仍要简洁。细节移到 `--help` 或外链技能。

### 5. 交叉引用其他技能

用技能名 + 明确标记：`**必需子技能：** Use superpowers:test-driven-development`  
**不要用 `@` 路径链到 SKILL.md** —— `@` 会强制预加载、烧上下文。

## 流程图使用

**仅当：** 决策不显然、可能过早停止的循环、「何时用 A vs B」。  
**不要用于：** 参考表、代码示例（用 Markdown）、线性步骤（用编号列表）、无语义标签（step1、helper2）。

流程图样式见 `graphviz-conventions.dot`。可用本目录 `render-graphs.js` 将技能中的图渲染为 SVG。

## 代码示例

**一个极好示例胜过许多平庸示例。** 选最相关语言；完整可运行；注释解释**为何**；来自真实场景。

## 技能的铁律（与 TDD 相同）

```
没有先失败的测试，就没有新技能
```

适用于**新技能**与**对现有技能的编辑**。

写技能前先写测试？删掉重来。无例外：不是「小改动例外」、不是「只加一节」、不是「文档更新例外」。

**必需背景：** superpowers:test-driven-development 解释为何如此。

## 按技能类型测试

- **纪律型：** 学术问题 + 压力场景 + 多重压力组合；成功标准：高压下仍遵守  
- **技术型：** 应用变体场景；成功标准：能在新场景正确应用技术  
- **模式型：** 识别/反例场景；成功标准：能判断何时适用  
- **参考型：** 检索与应用场景；成功标准：能找到并正确应用信息  

## 堵上合理化借口

纪律技能需显式封死漏洞：不只说规则，还列出「不要保留作参考、不要边写测试边改编」等；加原则「违反字面即违反精神」；建合理化表；列危险信号；在 description 中加入**即将违规**的症状词。

详见 `persuasion-principles.md`。

## 技能的 RED-GREEN-REFACTOR

- **红：** 无技能跑压力场景，逐字记录行为与借口  
- **绿：** 写最小技能只针对这些失败；再跑同场景应遵循  
- **重构：** 发现新借口 → 显式堵上 → 再测至「防弹」

完整方法论：`testing-skills-with-subagents.md`。

## 反模式

叙事式单次会话故事、多语言稀释示例、流程图里塞代码、无意义标签（helper1）等 —— 避免。

## 停：在换下一个技能之前

**写完任一技能后必须停下完成部署流程。** 不要无测试批量创建多个技能。

## 技能创建清单（TDD 适配）

**重要：** 用 TodoWrite 为下列每一项建待办。

**红：** 3+ 组合压力（纪律技能）；无技能跑场景并逐字记录基线  
**绿：** 命名/YAML/描述/关键词/概述/针对基线失败/内联或链文件/单语言好例；有技能跑场景确认遵循  
**重构：** 新借口、合理化表、危险信号、再测至防弹  
**质量：** 小流程图（仅当需要）、速查表、常见错误、无叙事流水账、支撑文件仅工具或重参考  
**部署：** 提交 git；若广泛有用可考虑贡献 PR  

（与英文原版逐项一一对应；部署前请对照英文清单补全细节。）

## 发现工作流

1. 遇到问题（「测试不稳定」）  
2. 找到 SKILL（description 匹配）  
3. 扫概述  
4. 读速查/模式  
5. 仅在实现时加载示例  

**为此优化** —— 可搜索词尽早、尽量多出现。

## 底线

**撰写技能 = 对流程文档做 TDD。**

同一铁律：无失败测试则无技能。同一循环：红 → 绿 → 重构。同一收益：更高质量、更少意外、更耐压力。

若你对代码遵循 TDD，对技能也应如此 —— 同一纪律，应用到文档。
