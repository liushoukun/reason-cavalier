# Skill 撰写最佳实践（Anthropic 官方指南 · 简体中文精要）

> **说明：** 仓库根目录 `skills/writing-skills/anthropic-best-practices.md` 为英文**全文**（含 Mintlify 图片与长示例）。本文件为同一文档结构的**简体中文精要译本**，便于中文读者快速掌握要点；实施细节、完整代码块与插图请以英文原文为准。

## 核心原则

### 简洁至上

上下文窗口是**公共品**。技能与系统提示、对话历史、其他技能元数据及用户请求共享上下文。启动时通常只预加载各技能的 `name`/`description`；`SKILL.md` 在相关时才加载。但一旦加载，`SKILL.md` 中每个 token 都与历史竞争——因此要**极省**。

**默认假设：** Claude 已很聪明。只补充其**不**已有的信息，并逐段问：「真需要解释吗？」「能假设它知道吗？」「这段值得花 token 吗？」

### 匹配「自由度」

- **高自由度**（纯文字步骤）：多种做法都成立、依赖情境、靠启发式 —— 如代码评审流程清单。  
- **中自由度**（伪代码/带参脚本）：存在首选模式、允许一定变化 —— 如报告生成模板函数。  
- **低自由度**（固定脚本、少参数）：操作脆弱、必须一致执行 —— 如数据库迁移命令行，**不得**改命令乱加 flag。  

类比：窄桥悬崖 = 低自由度给精确护栏；开阔无险 = 高自由度给方向即可。

### 在你将使用的所有模型上测试

技能效果依赖底层模型。Haiku 是否需要更多指引？Sonnet 是否清晰高效？Opus 是否被过度解释？跨模型使用时要折中。

## 技能结构（YAML）

`SKILL.md` 头需要 `name`（≤64 字符）与 `description`（≤1024 字符）。完整字段见 [agentskills.io/specification](https://agentskills.io/specification)。

### 命名

推荐**动名词**（verb+-ing）：如 Processing PDFs。也可用名词短语或动宾短语。避免 Helper/Utils 等泛名；同一库内命名风格一致。

### 撰写有效 description

**始终第三人称**（description 会进系统提示）。  
**具体 + 关键词**：说明**做什么**与**何时触发**（症状、场景）。  
**避免**第一/第二人称与空泛描述。

### 渐进披露

`SKILL.md` 作目录与入口；重内容拆到按需加载的附属文件。建议 `SKILL.md` 正文 <500 行；逼近则拆分。

**模式：** 高层指南 + 外链参考；按领域拆 `reference/*.md`；条件细节（简单默认写正文，高级特性链到专文）。

**避免深嵌引用：** 从 `SKILL.md` 出发**一层**链到参考文件最稳；链套链可能导致模型只 `head` 预览而不读全。

**长参考文件：** >100 行时在顶部加目录，便于部分阅读时仍见全貌。

## 工作流与反馈循环

复杂任务拆顺序步骤；可给**可复制勾选**清单。常见模式：**运行校验器 → 修错 → 重复**（脚本或 STYLE_GUIDE 充当校验器）。

## 内容规范

- **避免强时效表述**；若必须保留历史 API，用「当前方法」+ `<details>` 折叠旧版。  
- **术语一致**：全程同一词指同一概念（endpoint/field/extract 等不要混用）。

## 常见模式（节选）

- **默认进 `scripts/` 的 CLI 包装**：`scripts/` 可执行，`SKILL.md` 说明何时调用。  
- **捆绑资源**：`references/`、`assets/` 等大文件；`SKILL.md` 只写导航。  
- **多文件技能模板**：overview + 多参考 + 可选 `scripts/`。

## 评估与测量

定义技能成功指标；用模型对比前后；收集失败类型；迭代 description 与正文。

## 技术细节（运行时）

Agent Skills 架构：`SKILL.md` + 可选资源；工具可列目录、读文件、执行；模型自主决定加载深度。进展回调（如 `pre_tool_use`）可用于引导技能使用 —— 详见英文原文「Runtime environment」节。

## 技能创建检查清单（摘要）

- 初稿用最少指令；在目标模型上测试；与成功示例对照；读完整文件再判断；检查冗余与术语；验证所有链接；用第三方视角自审；部署后基于失败模式迭代。
