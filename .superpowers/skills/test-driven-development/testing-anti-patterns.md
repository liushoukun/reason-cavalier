# 测试反模式

**在以下情况加载本文：** 编写或修改测试、添加 mock，或忍不住想向生产类添加「仅测试用」方法时。

## 概述

测试必须验证**真实行为**，不是 mock 行为。mock 是隔离手段，不是被测对象。

**核心原则：** 测代码**做什么**，不要测 mock **做什么**。

**严格 TDD 能防止这些反模式。**

## 铁律

```
1. 绝不测试 mock 行为
2. 绝不在生产类上加仅测试用的方法
3. 绝不在不理解依赖的情况下 mock
```

## 反模式 1：测试 mock 行为

**违规：**
```typescript
// ❌ 差：在测 mock 是否存在
test('renders sidebar', () => {
  render(<Page />);
  expect(screen.getByTestId('sidebar-mock')).toBeInTheDocument();
});
```

**为何错：** 你在验证 mock 工作，不是组件工作；mock 在就过、不在就挂；与真实行为无关。

**人类搭档的纠正：**「我们是不是在测 mock 的行为？」

**修复：**
```typescript
// ✅ 好：测真实组件或不 mock
test('renders sidebar', () => {
  render(<Page />);
  expect(screen.getByRole('navigation')).toBeInTheDocument();
});
// 若必须 mock sidebar 以隔离：不要断言 mock —— 测 Page 在 sidebar 存在时的行为
```

### 闸门

```
在对任何 mock 元素断言之前：
  问：「我在测真实组件行为，还是仅 mock 存在？」

  若测 mock 存在：
    停下 —— 删断言或取消 mock

  改为测真实行为
```

## 反模式 2：生产代码中的仅测试方法

**违规：** 生产类有 `destroy()` 仅测试 `afterEach` 调用 —— 危险、混淆生命周期、违反 YAGNI。

**修复：** 把清理放进 `test-utils/` 的辅助函数，生产类不暴露测试专用 API。

### 闸门

```
在向生产类加任何方法之前：
  问：「是否仅测试使用？」

  若是：停下 —— 不要加；放测试工具里

  再问：「此类是否拥有该资源生命周期？」
  若否：停下 —— 方法放错类
```

## 反模式 3：不理解就 mock

**违规：** mock 掉测试仍依赖其副作用的方法（例如阻止写入配置），导致断言逻辑断裂。

**修复：** 在**正确层级** mock（只 mock 慢/外部部分），保留测试需要的副作用。

### 闸门

```
mock 任何方法之前：
  停下 —— 先别 mock

  1. 真实方法有哪些副作用？
  2. 本测试是否依赖其中任一副作用？
  3. 我是否完全理解本测试需要什么？

  若依赖副作用：在更低层 mock（真正慢/外部的操作）
  或使用保留必要行为的 test double
  不要 mock 测试还依赖的高层方法

  若不确定依赖：先用真实实现跑测试，观察必须发生什么，再在合适层级做最小 mock
```

## 反模式 4：不完整 mock

**违规：** 只 mock「你觉得需要」的字段，下游却访问未包含字段 → 静默失败、假信心。

**铁律：** mock **完整**的真实数据结构，而非仅当前测试立刻用到的字段。

**修复：** 对照文档/真实响应，包含下游可能消费的所有字段。

### 闸门

```
创建 mock 响应之前：
  查：真实 API 响应包含哪些字段？
  行动：
    1. 从文档/示例看真实响应
    2. 包含系统下游可能用到的**全部**字段
    3. 验证 mock 与真实 schema 完全一致

  关键：做 mock 就必须理解**整个**结构；部分 mock 在代码依赖被省略字段时会静默失败
  不确定：包含所有文档字段
```

## 反模式 5：集成测试当「后话」

**错：** 实现完了才说「可以测了」却没测试。  
**对：** TDD 循环：红 → 绿 → 重构 → **再**声称完成。

## Mock 过于复杂时

**警告：** mock 搭建比测试逻辑长；为通过而 mock 一切；mock 缺真实组件方法；一改 mock 测试就炸。  
**问：**「这里真的需要 mock 吗？」—— 真实组件的集成测试常比复杂 mock 更简单。

## TDD 如何防止这些

1. **先写测试** → 迫使你想清在测什么  
2. **看它失败** → 确认测的是真实行为而非 mock  
3. **最少实现** → 测试专用方法不易渗入生产  
4. **真实依赖** → mock 前你已看到测试真正需要什么  

**若在测 mock 行为，你已违反 TDD** —— 未先在真实代码上见失败就加了 mock。

## 速查

| 反模式 | 修复 |
|--------|------|
| 断言 mock 元素 | 测真实组件或取消 mock |
| 生产类测试专用方法 | 移到测试工具 |
| 不理解就 mock | 先理解依赖，最小 mock |
| 不完整 mock | 完整镜像真实 API |
| 测试当后话 | TDD：测试先行 |
| mock 过复杂 | 考虑集成测试 |

## 危险信号

- 断言查 `*-mock` test id  
- 方法仅被测试文件调用  
- mock 搭建占测试 >50%  
- 去掉 mock 测试就挂  
- 说不清为何需要 mock  
- 「为安全起见先 mock」  

## 底线

**Mock 是隔离工具，不是被测对象。**

若 TDD 暴露你在测 mock 行为，你已经走偏。

修复：测真实行为，或质疑是否应 mock。
